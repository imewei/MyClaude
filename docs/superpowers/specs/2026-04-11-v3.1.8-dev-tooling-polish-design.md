### Ordering rationale: A enables B

- **Commit A first** — replaces `interrogate` → `docstr-coverage`, dropping the archived `py` 1.11.0 transitive out of the lockfile. This retroactively makes PYSEC-2022-42969 vanish from `pip-audit` output. The observable win is the gate command output changing from `No known vulnerabilities found, 1 ignored` to `No known vulnerabilities found` (no ignores).
- **Commit B second** — adds the pip-audit CI workflow + `make audit` target. Both use bare `uv run pip-audit` with NO `--ignore-vuln` flag because Commit A has already cleaned the lockfile. If B were to land before A, it would have to ship with an inline `--ignore-vuln` flag that A would retroactively make unnecessary — an extra edit cycle avoided by the A-before-B ordering.
- **Commit C third** — trivial agent delegation polish. Independent of A and B but lands after them by convention.
- **Commit D last** — version bump reflects the committed tree.

### Item 2 — pip-audit CI wiring (Commit B)

**Files (2)**:
1. `Makefile` — add new `audit:` target
2. `.github/workflows/audit.yml` — new file

**`Makefile`** — add a new `##@ Security` section after the existing `##@ Quality` section. (Confirmed via grep: the current Makefile has sections General, Cleaning, Documentation, Development, Quality — no Testing section, so no ambiguity about where to insert.) Content:

```makefile
##@ Security

audit: ## Run pip-audit on dev dependencies
	@echo "Running pip-audit on dev dependencies..."
	@uv run pip-audit
	@echo "  ✓ pip-audit complete — no known vulnerabilities"
```

No `--ignore-vuln` flag. Commit A has already resolved PYSEC-2022-42969 by dropping `interrogate`, so the bare invocation runs clean.

**`.github/workflows/audit.yml`** — new file, ~35 lines. Mirrors the shape of the existing `docs.yml` workflow:

```yaml
name: pip-audit

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]
    paths:
      - 'pyproject.toml'
      - 'uv.lock'
      - '.github/workflows/audit.yml'

jobs:
  pip-audit:
    name: Dependency CVE scan
    runs-on: ubuntu-latest
    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Install uv
        uses: astral-sh/setup-uv@v4
        with:
          version: "latest"

      - name: Install dev dependencies
        run: uv sync

      - name: Run pip-audit
        run: uv run pip-audit
```

Notes on the workflow design:
- **Triggers**: runs on push to `main` and on PRs that modify `pyproject.toml`, `uv.lock`, or the workflow file itself. This keeps CI cheap — it only re-audits when dependencies change.
- **`astral-sh/setup-uv@v4`** is the canonical GitHub Action for installing uv. Match whatever version the existing `docs.yml` uses (spot-check during Commit B Step 1).
- **`uv sync`** installs all dependencies (including dev group with `docstr-coverage` and `pip-audit` from Commit A).
- **`uv run pip-audit`** — bare invocation, no flags. Fails the job on any HIGH/CRITICAL finding.
- **No `continue-on-error`** — we want the workflow to block merge on vulnerabilities.

**Commit B verification**: after landing, monitor the first CI run on the commit itself (the workflow triggers on push-to-main, so it runs immediately). Expected: runs green, reports `No known vulnerabilities found`, exits 0. If it fails, diagnose before continuing.

**Budget impact**: none.

## Rollback plan

**R1 — Commit A factual error after landing** (e.g., `docstr-coverage --fail-under` flag form turns out to be different in latest): `git revert <A-sha>`. Restores `interrogate`, re-introduces PYSEC-2022-42969 into the `--ignore-vuln` escape valve, but leaves the workflow in a known-working state. Cost: one revert commit.

**R2 — Commit B CI workflow fails on first real PR** (subtle path-filter bug or setup-uv version mismatch): either `git revert <B-sha>` OR fix forward with a targeted `fix(ci): ...` commit. Fix-forward is usually cleaner for CI workflows.

**R3 — Commit A breaks `docstr-coverage` on `tools/`** because the default exclusion rules differ from `interrogate`'s:
- First fix attempt: add explicit exclusion flags to match `interrogate`'s defaults (e.g., `--exclude tests/`, `--skip-init`). If a simple flag restores the 80% threshold, fold it into Commit A's diff before landing.
- Second fix attempt: revert A and escalate.

**R4 — Commit D (version bump) lands with a mismatched note or version string**: trivial fix-forward edit, no revert needed.

**R5 — `pip-audit` surfaces a NEW HIGH CVE during Commit A's gate** (unrelated to `interrogate` / `py`): stop the release. Either (a) bump the offending dep if a fix exists, (b) add a new `--ignore-vuln` with justification (reintroducing the escape-valve pattern, spec amendment required), or (c) defer the entire v3.1.8 release and ship a tooling-only CVE-fix commit first.

**R6 — Context7 unreachable at commit authoring time** for the `docstr-coverage` or GitHub Actions lookup: fall back to WebFetch on the upstream repo README. If both sources are down, mark the relevant claim `[unverified — Context7 and WebFetch both unreachable at <date>]` and ship — same defensive pattern used in v3.1.5 / v3.1.6 / v3.1.7.
