## Release architecture

### Commit sequence (4 commits)

```
Commit B  feat(science-suite): Julia → Python handoff for nonlinear time-series tools
Commit C  feat(science-suite): BAR free-energy worked example bridging Langevin and non-eq theory
Commit D  feat(science-suite): freud ecosystem for physical correlations
Commit E  chore(core): bump all plugins to v3.1.6
```

Internal IDs B / C / D map to items B / C / D from the brainstorm — easy to trace in git log. No polish commit A this time (unlike v3.1.5) since v3.1.6 has no tooling work. Version bump lands last to reflect the actually-shipped tree.

### Ordering rationale: B → C → D

Lowest-risk first:

- **B** is lowest-risk because all Python packages it names (`nolds`, `antropy`, `IDTxl`, `pyEDM`, `pyunicorn`, `teaspoon`) are already documented in existing v3.1.4 skills — v3.1.6 only adds the `PythonCall.jl` handoff idiom, no new package verification load. Ships first for validator-green momentum.
- **C** is medium-risk because `pymbar` has moved its BAR entry point at least twice historically (top-level → `pymbar.other_estimators.bar`). Requires Context7 verification of the current API.
- **D** is highest-risk because `freud` has the largest new API surface in v3.1.6: `freud.density.RDF`, `freud.diffraction.StaticStructureFactorDebye` / `StaticStructureFactorDirect`, `freud.order.Steinhardt`, `freud.order.Nematic`, `freud.density.IntermediateScattering`. Five separate API claims, each a drift risk.

A bisect against any future regression terminates on the right commit, not a speculative one.

### Pre-execution verification (just-in-time, per commit)

Each content commit runs Context7 lookups **immediately before authoring**, not as a batch upfront. Just-in-time keeps verification tied to the commit that actually needs it and avoids wasting context on packages whose commit might get deferred.

**Before Commit B**: resolve `PythonCall.jl` via Context7; fetch canonical `pyimport` / `@py` macro pattern. Fallback: WebFetch `github.com/JuliaPy/PythonCall.jl` README.

**Before Commit C**: resolve `pymbar` via Context7; verify current location of BAR (`pymbar.BAR` vs `pymbar.other_estimators.bar`). Secondary: resolve `alchemlyb` (mentioned but not example-depth).

**Before Commit D**: resolve `freud` via Context7; verify five API entry points (RDF, StaticStructureFactorDebye / Direct, Steinhardt, Nematic, IntermediateScattering). Fallback: WebFetch `github.com/glotzerlab/freud` README for any API that can't be verified.

**Verification gate**: if any lookup contradicts mental model, pause authoring and surface the discrepancy. If a claim cannot be verified at authoring time, tag it `[unverified — see upstream docs]` (same defensive pattern v3.1.5's Commit C subagent used for `ACEpotentials.jl` v0.10 symbols).

## Rollback plan

Per-commit rollback granularity is the whole point of the 4-commit split.

**R1 — Commit C factual error surfaces after landing but before E**: revert C only; B and D stay. Fix the factual error in a new content commit, re-run validator gate, re-sequence before the version bump.

**R2 — Validator gate fails mid-commit**: the commit is not staged yet (validator runs before `git commit`), so there's nothing to roll back. Diagnose, fix the working tree, re-run the gate. Same loop as v3.1.5.

**R3 — Post-landing regression discovered after Commit E**: revert individual content commit(s), keep E intact, ship a fast-follow `chore(core): revert to v3.1.5 state` if the release note needs to match.

**R4 — Upstream package drift discovered post-landing** (e.g., reader reports `pymbar.other_estimators.bar` no longer works because Chodera lab moved it *again*): treat as a v3.1.7 patch — amend the offending file, update the `[verified YYYY-MM]` tag, ship as a tiny `fix(science-suite)` commit. Do not revert v3.1.6; patch forward.

**R5 — Context7 returns stale docs**: fallback to WebFetch on the canonical GitHub README. If even that contradicts, mark the relevant claim `[unverified]` with an inline comment and ship anyway — same defensive pattern v3.1.5 used for `ACEpotentials.jl` v0.10.
