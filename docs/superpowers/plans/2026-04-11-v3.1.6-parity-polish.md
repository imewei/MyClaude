## Calling Python nonlinear-dynamics tools from Julia via PythonCall.jl

When the surrounding codebase is Julia-first but the computation you need lives in `nolds` / `antropy` / `IDTxl` / `pyEDM` / `pyunicorn` / `teaspoon` — and no native Julia equivalent exists — use `PythonCall.jl`. This is the same package whose Python-facing half (`juliacall`) is already documented in `bifurcation-analysis` for the Python → Julia direction. Together they give a coherent Julia ↔ Python bridge for both directions.

### Canonical import pattern

```julia
using PythonCall

# One-time setup: CondaPkg.jl manages the Python environment
# pkg> add CondaPkg
# julia> CondaPkg.add("nolds")
# julia> CondaPkg.add("antropy")

nolds   = pyimport("nolds")
antropy = pyimport("antropy")
idtxl   = pyimport("idtxl")            # via IDTxl's top-level package
```

### Concrete example — compute `lyap_r` from Julia

```julia
using PythonCall
nolds = pyimport("nolds")

# Julia-side time series — lorenz-ish dynamics, shape (N,)
N  = 10_000
x  = [0.0 for _ in 1:N]
for t in 2:N
    x[t] = 3.9 * x[t-1] * (1 - x[t-1])    # logistic map, chaotic regime
end

# Marshal Julia vector -> NumPy array -> call nolds
py_x  = Py(x)                               # PythonCall does the conversion
lyap  = pyconvert(Float64, nolds.lyap_r(py_x, emb_dim=5, lag=1))
println("Largest Lyapunov exponent ≈ $(lyap)")
```

The same pattern works for `antropy.perm_entropy`, `antropy.higuchi_fd`, `idtxl` transfer-entropy calls, `pyEDM.Simplex`, `pyunicorn.RecurrencePlot`, and the rest of the Python nonlinear-dynamics ecosystem documented in the Python counterparts table above.

### Caveats

- **Array marshaling cost** — each `Py(x)` call copies the Julia array to Python-owned memory. For short calls this is noise; for a million-point recurrence plot, prefer calling the Python function once over the full series rather than per-window.
- **GIL** — PythonCall.jl holds the GIL during each Python call. Multithreaded Julia code (`@threads`) that calls into Python will serialize on the GIL, making it no faster than single-threaded. For parallel workloads, dispatch entire independent problems via `Distributed.jl` workers (each worker gets its own Python interpreter).
- **`PyCall.jl` vs `PythonCall.jl`** — `PyCall.jl` is the legacy package from ~2015, still widely used in older code. New work should prefer `PythonCall.jl` because it ships `CondaPkg.jl` for deterministic Python-environment management and pairs with `juliacall` (see `bifurcation-analysis` for the reciprocal Python → Julia direction).
- **Startup cost** — the first `pyimport` call in a session incurs a ~1-2s Python interpreter boot. Subsequent calls are fast. For ad-hoc scripts, this is tolerable; for latency-sensitive workflows, keep a single long-lived session.

### Cross-reference

The reciprocal direction (Python → Julia via `juliacall`, for bifurcation continuation through BifurcationKit.jl) is documented in `bifurcation-analysis` under "Python escape hatch". Together, the two handoff sections cover both halves of the Julia ↔ Python nonlinear-dynamics interop story.
````

## Worked example — BAR from Langevin work samples

End-to-end bridging of the Langevin ensemble pattern from `stochastic-dynamics` with the BAR free-energy estimator. The Jarzynski equality `⟨exp(-βW)⟩ = exp(-βΔF)` gives an unbiased free-energy estimate from non-equilibrium work samples, but naive exponential averaging has catastrophic variance — the estimate is dominated by rare low-work trajectories. Bennett Acceptance Ratio (BAR) uses both forward and reverse work distributions to achieve the minimum-variance estimator.

### When to prefer BAR vs Jarzynski vs MBAR

| Situation | Use |
|-----------|-----|
| Only forward-protocol work samples available | Naive Jarzynski `⟨exp(-βW)⟩` (high variance — use only for validation) |
| Forward + reverse protocols on two thermodynamic states | **BAR** (optimal two-state estimator) |
| Multiple thermodynamic states (alchemical ladder, λ-windows) | **MBAR** — minimum-variance multistate generalization |

### Stage 1 — Generate forward and reverse work samples

A non-equilibrium switching protocol varies a control parameter λ from 0 to 1 (forward) or 1 to 0 (reverse) at a finite rate. Along each trajectory, accumulate the work `W = ∫ (∂H/∂λ) dλ̇ dt`. Use the Langevin ensemble pattern from `stochastic-dynamics` to generate N independent forward and reverse trajectories.
```

```python
import jax, jax.numpy as jnp, jax.random as jr

# Switching Hamiltonian: H(x; λ) = (1-λ) * U_A(x) + λ * U_B(x)
# Example: morph between two 1D harmonic oscillators with different minima.
def U_A(x): return 0.5 * x**2                         # state A: minimum at 0
def U_B(x): return 0.5 * (x - 2.0)**2                 # state B: minimum at 2
def H(x, lam): return (1 - lam) * U_A(x) + lam * U_B(x)
dH_dlam = lambda x: U_B(x) - U_A(x)                    # ∂H/∂λ, λ-independent for linear interpolation

def switch_langevin(key, x0, lam0, lam1, n_steps, dt, kT, gamma=1.0):
    """Integrate overdamped Langevin along a linear λ schedule; accumulate work."""
    grad_H = lambda x, lam: jax.grad(lambda x_: H(x_, lam))(x_)
    def body(carry, key_t):
        x, W = carry
        t_idx, subkey = key_t
        lam = lam0 + (lam1 - lam0) * (t_idx / n_steps)
        # Accumulate work: dW = (∂H/∂λ)(x) * dλ
        dW = dH_dlam(x) * ((lam1 - lam0) / n_steps)
        # Langevin step at current λ
        force = -jax.grad(lambda x_: H(x_, lam))(x)
        noise = jr.normal(subkey, ())
        x_next = x + (force / gamma) * dt + jnp.sqrt(2 * kT * dt / gamma) * noise
        return (x_next, W + dW), None
    keys = jr.split(key, n_steps)
    (x_final, W_total), _ = jax.lax.scan(body, (x0, 0.0), (jnp.arange(n_steps), keys))
    return W_total

# Generate N forward and N reverse work samples
N, kT, dt, n_steps = 1000, 1.0, 0.01, 1000
keys_F = jr.split(jr.PRNGKey(0), N)
keys_R = jr.split(jr.PRNGKey(1), N)
W_F = jax.vmap(lambda k: switch_langevin(k, 0.0, 0.0, 1.0, n_steps, dt, kT))(keys_F)   # 0 -> 1
W_R = jax.vmap(lambda k: switch_langevin(k, 2.0, 1.0, 0.0, n_steps, dt, kT))(keys_R)   # 1 -> 0 (note: started from state B minimum)
```

### Stage 3 — Variance comparison with naive Jarzynski

Naive Jarzynski uses only the forward samples and computes `ΔF = -kT log⟨exp(-βW_F)⟩`. The estimator is unbiased but its variance scales as `exp(2 σ²_W / kT²)` — an order-of-magnitude increase in work spread blows up the variance by orders of magnitude.

```python
import numpy as np

def jarzynski_naive(W_F, kT):
    return -kT * np.log(np.mean(np.exp(-W_F / kT)))

def jarzynski_cumulant(W_F, kT):
    """Second-cumulant Jarzynski (Gaussian-work approximation)."""
    return np.mean(W_F) - 0.5 * np.var(W_F) / kT

# Subsample the forward data to study convergence
for M in [50, 100, 250, 500, 1000]:
    idx = np.random.choice(len(W_F), M, replace=False)
    dF_naive = jarzynski_naive(W_F[idx], kT)
    dF_cum   = jarzynski_cumulant(W_F[idx], kT)
    print(f"N={M:4d}  naive={dF_naive:+.4f}  cumulant={dF_cum:+.4f}  BAR_same_N={bar(W_F[idx]/kT, W_R[idx]/kT)['Delta_f']*kT:+.4f}")
```

The naive Jarzynski estimate converges slowly and is biased for small N; BAR is already close to the true value at N=50. This is the single strongest argument for using BAR whenever both forward and reverse protocols are feasible.

## Python `freud` ecosystem — the soft-matter reference toolkit

The Glotzer group's `freud` library (University of Michigan) is the production tool for physical-system correlation analysis. No native Julia equivalent exists — Julia users go through `PythonCall.jl` (see `chaos-attractors` for the canonical handoff pattern). `freud` covers RDF, structure factors, bond-orientational order, intermediate scattering, and all the other two-point and higher-order correlations that soft-matter analysis needs.

### Why freud

- **Periodic boundary handling** correct for triclinic boxes and arbitrary cell geometries
- **O(N log N) neighbor lists** via cell-list construction, reusable across multiple analyses on the same frame
- **GPU acceleration** via the CuPy backend for large systems
- **Trajectory reader integration** with MDAnalysis and MDTraj (both of which expose frame-iteration protocols `freud` consumes directly)
- **Minimal dependencies** — NumPy + optional CuPy; installable via `pip install freud-analysis`

### Radial distribution function g(r)

The canonical two-point correlation in an isotropic system.

```python
import freud
import numpy as np

# Example: a single frame with N particles in a cubic box of side L
box      = freud.box.Box.cube(L=10.0)
points   = np.random.uniform(-5, 5, size=(1000, 3))        # shape (N, 3)

rdf = freud.density.RDF(bins=100, r_max=4.5)
rdf.compute(system=(box, points))

# Plot g(r)
import matplotlib.pyplot as plt
plt.plot(rdf.bin_centers, rdf.rdf)
plt.xlabel("r"); plt.ylabel("g(r)")
```

The `compute()` call can be chained over multiple frames via `reset=False` to accumulate statistics — the correct pattern for a trajectory average. Cross-check g(r) against the analytical Ornstein-Zernike closure from `correlation-math-foundations` for dilute hard-sphere or Lennard-Jones references.

### Static structure factor S(q)

Two implementations differ in how they handle the discrete q-space grid:

- **`freud.diffraction.StaticStructureFactorDebye(bins, k_max, k_min)`** — Debye scattering formula, O(N²) but works for any geometry including non-periodic systems
- **`freud.diffraction.StaticStructureFactorDirect(bins, k_max, k_min)`** — direct reciprocal-space summation, requires periodic box, O(N · N_q), faster for dense systems

```python
sf = freud.diffraction.StaticStructureFactorDirect(bins=100, k_max=10.0, k_min=0.5)
sf.compute(system=(box, points))
plt.plot(sf.bin_centers, sf.S_k)
```

Cross-check S(q) against `g(r)` via the Fourier transform of `h(r) = g(r) - 1`; the `correlation-math-foundations` skill has the Wiener-Khinchin pair.

### Bond-orientational order parameters

# Steinhardt Q_l order parameters — classic crystalline / hexatic order

### Intermediate scattering F(q, t)

The dynamical counterpart to S(q) — the key observable for glass relaxation, dynamic heterogeneity, and alpha-process analysis.

```python
# Compute F(q, t) across a trajectory
isf = freud.density.IntermediateScattering(bins=50, k_max=8.0)
for frame_idx, (box_f, positions_f) in enumerate(trajectory):
    isf.compute(system=(box_f, positions_f), reset=(frame_idx == 0))
# isf.F_kt is shape (n_frames, n_q_bins)
```

Use `F(q*, t)` at the peak of `S(q)` (the structural-length-scale q*) as the alpha relaxation probe. Fit a stretched exponential `exp(-(t/τ)^β)` to extract the alpha relaxation time τ.

### Calling freud from Julia via PythonCall.jl

Because no native Julia equivalent exists, Julia users access `freud` through the `PythonCall.jl` handoff pattern documented in `chaos-attractors`:

```julia
using PythonCall
freud = pyimport("freud")
np    = pyimport("numpy")

box    = freud.box.Box.cube(L=10.0)
points = np.random.uniform(-5, 5, size=pytuple((1000, 3)))

rdf = freud.density.RDF(bins=100, r_max=4.5)

r_bins = pyconvert(Vector{Float64}, rdf.bin_centers)
g_of_r = pyconvert(Vector{Float64}, rdf.rdf)

The `pytuple` wrapping is necessary because `freud` expects a Python tuple `(box, points)` for the `system` keyword argument and `PythonCall.jl` does not implicitly convert Julia tuples.

### Caveats

- **No native Julia equivalent** — the only Julia path for production soft-matter correlation analysis is `PythonCall.jl` → `freud`. There are small pedagogical Julia packages for g(r) on a single frame, but none matches `freud`'s neighbor-list infrastructure, trajectory integration, or GPU support.
- **GPU path is CuPy-specific** — AMD ROCm / JAX / PyTorch are not supported by `freud`'s current GPU backend. If the surrounding pipeline is JAX, plan for a host-side handoff.
- **Trajectory readers** — `freud` does not read trajectory files itself; use `MDAnalysis` or `MDTraj` to iterate frames and pass `(box, positions)` tuples to `freud` per frame.
- **Box conventions** — `freud.box.Box` uses a specific sign convention for triclinic boxes (the `xy`, `xz`, `yz` tilt factors) that differs from LAMMPS and HOOMD defaults. Double-check the box construction when reading data from those tools.

## `freud` — algorithmic notes

The Glotzer group's `freud` library is the production tool for trajectory correlation analysis. Its internals are worth knowing because the same neighbor-list infrastructure is reused across every analysis (RDF, structure factor, Steinhardt, intermediate scattering) — building it once per frame amortizes the cost.

### Neighbor-list construction

`freud` uses a cell-list under the hood with O(N) build cost and O(N log N) query cost. The list supports:

- **Periodic boundary handling** for triclinic boxes (correct minimum-image convention on any cell geometry, not just orthogonal)
- **Reuse across analyses** — build the list once with `freud.locality.AABBQuery` and pass it to downstream analyzers via the `neighbors` keyword
- **k-nearest-neighbor queries** with `{"mode": "nearest", "num_neighbors": K}` for Steinhardt / bond-orientational analyses
- **Cutoff-radius queries** with `{"mode": "ball", "r_max": R}` for RDF / correlation-type analyses

```python
# Reuse a single neighbor list across multiple analyses
query = freud.locality.AABBQuery(box, points)
q4   = freud.order.Steinhardt(l=4); q4.compute(system=query, neighbors={"mode": "nearest", "num_neighbors": 12})
q6   = freud.order.Steinhardt(l=6); q6.compute(system=query, neighbors={"mode": "nearest", "num_neighbors": 12})
```

### Multi-frame averaging

Across a trajectory, pass `reset=False` to accumulate statistics without zeroing the histogram:

```python
rdf = freud.density.RDF(bins=100, r_max=5.0)
for frame_idx, (box_f, points_f) in enumerate(trajectory):
    rdf.compute(system=(box_f, points_f), reset=(frame_idx == 0))
# rdf.rdf is now the trajectory-averaged g(r)
```

### GPU acceleration

`freud` supports a CuPy backend for neighbor-list construction and the heavier analyzers. Enable via the `freud.parallel.set_num_threads()` CPU path or by passing CuPy arrays to the analyzers directly. The speedup is noticeable at N ≥ 10⁴; below that, the CPU path is faster due to GPU launch overhead.

### Comparison with alternatives

| Tool | Strength | Weakness |
|------|----------|----------|
| **`freud`** | Fastest single-frame analysis, best neighbor-list infrastructure, GPU-ready | Does not read trajectory files itself |
| **`MDAnalysis`** | Best trajectory-reader integration, broadest analysis catalog (ACF, hydrogen bonds, PCA) | Slower per-frame than `freud` for pure correlation tasks |
| **`MDTraj`** | Fast frame iteration, Keras-compatible for ML workflows, huge format support | Correlation analyses less comprehensive than `freud` |

Typical production pattern: use `MDAnalysis` or `MDTraj` as the frame iterator, pass each frame's `(box, positions)` tuple to `freud` for the correlation analysis.

See `correlation-physical-systems` for the "what to compute" side (RDF, S(q), Steinhardt, F(q,t) worked examples).
````
