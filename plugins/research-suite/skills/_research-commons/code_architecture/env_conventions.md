# Environment conventions

Every code artifact produced by the research-spark stack uses the same environment contract.

## Python version

Python 3.12 or later. Skills should generate `pyproject.toml` with `requires-python = ">=3.12"`.

## Dependency management

Use `uv` for dependency resolution and virtual environment management. The skill-emitted repos ship with:
- `pyproject.toml` (declarations)
- `uv.lock` (pinned resolution, committed)
- `.python-version` (one line, Python version)

Commands the user will run (document these in the emitted README):
```
uv sync               # install
uv run pytest         # test
uv run python -m mymod  # execute
```

Do not emit `requirements.txt` or `environment.yml`.

## pyproject.toml layout

```toml
[project]
name = "my-prototype"
version = "0.1.0"
description = "Generated by numerical-prototype for <idea-slug>"
requires-python = ">=3.12"
dependencies = [
    "jax>=0.4.30",
    "jaxlib>=0.4.30",
    "numpy>=2.0",
    "scipy>=1.14",
]

[project.optional-dependencies]
dev = [
    "pytest>=8.0",
    "pytest-xdist>=3.5",
    "hypothesis>=6.100",
    "ruff>=0.5",
]
test = ["pytest>=8.0", "pytest-xdist>=3.5", "hypothesis>=6.100"]

[build-system]
requires = ["hatchling"]
build-backend = "hatchling.build"

[tool.ruff]
line-length = 100
target-version = "py312"

[tool.pytest.ini_options]
testpaths = ["tests"]
addopts = "-ra -q"
```

## Linting and formatting

Use `ruff` for both linting and formatting. Do not use black + flake8 + isort separately; ruff covers all three.

## GPU/accelerator handling

JAX installs are platform-specific. In `pyproject.toml`, list `jax>=x.y.z` but let `uv` resolve the platform. Document GPU-specific installation in the emitted README rather than baking it into dependencies.

## Why these rules

Consistency with `homodyne` and `heterodyne`, which already use this pattern. Consistency across the stack means one install recipe works for everything Wei emits from research-spark.
