# Repository layout

Generated code repos follow this structure. It matches the `homodyne`, `heterodyne`, and `RheoJax` pattern so that code from the research-spark stack drops into familiar scaffolding.

```
<repo-root>/
├── pyproject.toml
├── uv.lock
├── .python-version
├── README.md                   # how to install and run
├── LICENSE                     # match lab default (BSD-3 unless told otherwise)
├── .gitignore
├── src/
│   └── <package_name>/
│       ├── __init__.py
│       ├── core.py             # state and step function
│       ├── observables.py      # predicted-observable extractors
│       ├── integrators.py      # time-integration strategies
│       ├── params.py           # Params dataclass, validation
│       └── reference.py        # analytic reference cases for limit tests
├── tests/
│   ├── test_core.py
│   ├── test_observables.py
│   ├── test_invariants.py
│   ├── test_limits.py
│   └── benchmarks/
├── examples/
│   ├── run_minimal.py          # minimal reproducible example
│   └── convergence_study.ipynb # or .py if notebooks are not preferred
└── docs/
    ├── derivation.md           # brief prose derivation (symlink to 04_theory.md)
    └── predicted_observable.md # symlink to Stage 6 artifact
```

## Naming

- Package name: `<idea_slug>` with underscores (e.g., `rheox_spectral_gap`)
- Repo directory: `<idea-slug>` with hyphens (e.g., `rheox-spectral-gap`)

## README contents

A minimal README generated by numerical-prototype includes:
1. One-paragraph description linking back to `04_theory.md` and `06_prototype.md`
2. Install: `uv sync`
3. Test: `uv run pytest`
4. Minimal reproducible example: `uv run python examples/run_minimal.py`
5. Link to the predicted observable file in `docs/`

## What NOT to include

- `Makefile` (use `uv` commands directly)
- `setup.py` or `setup.cfg` (pyproject.toml only)
- `requirements.txt` (uv.lock only)
- Notebooks for production code (use `.py` files with `# %%` cell markers for notebook-style)

## Symlinks between skill artifacts and the repo

The `numerical-prototype` skill, when emitting a code repo, sets up symlinks in `docs/` to the Stage 4 theory artifact and the Stage 6 prototype artifact. This keeps the code honest: the derivation it claims to implement is visible from the repo.
